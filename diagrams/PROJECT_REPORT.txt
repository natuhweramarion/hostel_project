================================================================================
                               ABSTRACT
================================================================================

Hostel information management at institutions of higher learning has
traditionally been a manual, paper-based process prone to errors, delays, and
inefficiencies. This project presents the design and development of an Online
Hostel Information Management System (HMS) for Cavendish University Uganda — a
web-based platform built using the Django framework (Python) that automates
student accommodation registration, room allocation, payment tracking, and
administrative reporting.

The system addresses key challenges identified in the current manual process,
including inaccessibility of hostel availability information, slow room
allocation, lack of payment audit trails, and difficulty in generating
occupancy reports. The system provides two primary user roles: Students, who
can browse hostel listings with pricing and room type information, submit
booking requests, upload payment records, and receive printable receipts; and
Hostel Managers, who can manage hostel and room inventory, process allocations,
verify payments, review booking requests, and generate comprehensive reports
exportable to CSV.

The Agile (Scrum) development methodology was adopted, enabling iterative
development across three two-week sprints. The system was built on a three-tier
architecture comprising a Bootstrap 5.3 presentation layer, a Django 5.x
business logic layer organised into five application modules (users, hostels,
allocations, payments, reports), and a SQLite data layer managed through
Django's ORM. Key security features include CSRF protection, session-based
authentication, role-based access control, and POST-only logout.

System testing confirmed that all core functional requirements were met.
The Online HMS significantly reduces administrative burden, improves
transparency of the allocation process, and provides students with real-time
access to hostel information — ultimately contributing to a better student
accommodation experience at Cavendish University Uganda.

Keywords: Hostel Management System, Django, Room Allocation, Web Application,
          Student Accommodation, Information System, Agile Development

1.1 BACKGROUND OF THE STUDY
------------------------------

The management of student accommodation in universities and institutions of
higher learning is a critical administrative function that directly affects
student welfare, academic performance, and institutional efficiency. As
universities across Africa continue to experience growing student enrolment,
the challenges associated with hostel management have become increasingly
complex and demanding.

Cavendish University Uganda, located along Ggaba Road in Nsambya, Kampala,
is a private university offering a range of undergraduate and postgraduate
programmes. Like many institutions in Uganda, the university provides hostel
accommodation to a significant proportion of its students. The effective
management of this accommodation is essential not only for operational
efficiency but also for ensuring a conducive learning environment.

Traditionally, hostel management at many Ugandan universities has been
conducted manually — through physical ledgers, spreadsheets, and face-to-face
interactions. Students seeking accommodation are required to physically visit
the hostel office, fill in paper forms, make payments at the bank, and return
with physical receipts for manual verification. Room assignments are typically
recorded in registers, and occupancy information is not readily accessible
without consulting the hostel administrator.

This manual approach presents numerous challenges. Information about available
rooms, pricing, and hostel facilities is not easily accessible to students
before they arrive at the office. The allocation process is slow and
susceptible to human error. Payment records can be lost or misplaced.
Generating reports on occupancy, revenue, and allocation history requires
manually compiling data from multiple registers — a time-consuming and
error-prone process.

The rapid advancement of Information and Communication Technology (ICT) has
created an unprecedented opportunity to automate and streamline administrative
functions in universities. Web-based information systems have proven
particularly effective in transforming manual processes into efficient,
transparent, and accessible digital workflows. Many universities worldwide have
adopted online hostel management systems that allow students to browse
accommodation options, submit booking requests, make and track payments, and
receive electronic confirmations — all from the convenience of a web browser.

This project therefore sets out to design and develop an Online Hostel
Information Management System for Cavendish University Uganda — a web-based
platform that automates the hostel registration, allocation, payment tracking,
and reporting processes, thereby addressing the inefficiencies of the current
manual system.

The system was developed using Python and the Django web framework — a
high-level, batteries-included framework that follows the Model-View-Template
(MVT) architectural pattern. The frontend was designed using Bootstrap 5.3 for
a responsive, modern user interface accessible from any device.


1.2 PROBLEM STATEMENT
-----------------------

The current hostel management process at Cavendish University Uganda relies
heavily on manual, paper-based methods. This has resulted in the following
specific problems:

(i) Information Inaccessibility: Students cannot access up-to-date information
    about available rooms, pricing, hostel facilities, or room types without
    physically visiting the hostel office. This creates unnecessary travel,
    time wastage, and frustration, especially for students who are not yet
    on campus.

(ii) Slow and Error-Prone Allocation: Room allocation is performed manually
     by hostel staff using physical registers. This process is slow,
     susceptible to double-booking errors, and lacks an audit trail.
     There is no system-enforced constraint preventing a student from being
     allocated to more than one room simultaneously.

(iii) Poor Payment Tracking: Students submit payment receipts physically at
      the hostel office. Payment records are kept in paper registers that can
      be lost, damaged, or altered. There is no digital trail of which
      administrator verified a payment, when it was verified, or its current
      status.

(iv) Lack of Reporting Capability: Generating reports on hostel occupancy,
     payment revenue, and allocation history requires manually compiling data
     from registers. This makes it difficult for management to make timely,
     data-driven decisions about accommodation capacity and revenue.

(v) No Booking Request Workflow: Students have no formal mechanism to express
    a preference for a particular hostel before a room is assigned. Allocation
    decisions are made entirely by the hostel office without any student input.

These problems collectively reduce the quality of the student accommodation
experience, increase the administrative workload of hostel staff, and impede
management's ability to oversee hostel operations effectively.


1.3 OBJECTIVES OF THE STUDY
-----------------------------

1.3.1 Main Objective
The main objective of this study is to design and develop an Online Hostel
Information Management System that automates the hostel allocation, payment,
and reporting processes at Cavendish University Uganda.

1.3.2 Specific Objectives
The specific objectives of the study are:

(i)   To analyse the existing manual hostel management process and identify
      specific requirements for an automated system.

(ii)  To design a relational database schema that accurately models the
      hostel domain including hostels, blocks, rooms, allocations, payments,
      and booking requests.

(iii) To develop a web-based system with role-based access control that
      supports student self-service (browsing, booking, payment submission)
      and administrative functions (allocation, verification, reporting).

(iv)  To implement a secure payment tracking module that allows students to
      submit payment records with proof of payment and enables administrators
      to verify or reject submissions with a complete audit trail.

(v)   To design and implement a reporting module that generates real-time
      occupancy, allocation, and payment reports exportable to CSV format.

(vi)  To test the developed system against functional and non-functional
      requirements and evaluate its performance and usability.

This study is guided by the following research questions:

(i)   What are the specific challenges and inefficiencies in the current manual

      over-allocation of rooms and duplicate payment reference numbers?

(v)   To what extent does the developed system meet the functional and
      non-functional requirements identified during analysis?

This study covers the design, development, and testing of an Online Hostel

Included:
- Student user registration, login, and profile management
- Public hostel information page with room types, pricing, and facilities
- Student booking request submission and tracking
- Administrator room allocation management
- Student payment record submission with receipt upload
- Administrator payment verification workflow
- Printable payment receipt generation
- Hostel occupancy, allocation, and payment reporting with CSV export

- Automated room allocation algorithms (allocation is performed manually by

- Integration with the university's student information system (SIS)

                   CHAPTER TWO: LITERATURE REVIEW

2.1 INTRODUCTION
-----------------

This chapter reviews existing literature relevant to the development of an
Online Hostel Information Management System. The review covers the theoretical
foundations of management information systems, the evolution of hostel
management systems globally and within the African context, existing related
systems, and the identification of gaps that this study seeks to address.


2.2 OVERVIEW OF MANAGEMENT INFORMATION SYSTEMS
-------------------------------------------------

A Management Information System (MIS) is an organised combination of people,
hardware, communication networks, data sources, and policies and procedures
that stores, retrieves, transforms, and disseminates information in an
organisation (Laudon & Laudon, 2020). MIS are designed to support management
decision-making and provide a structured flow of information from operational
levels to management.

Stair and Reynolds (2018) define an information system as a set of
interrelated components that collect, process, store, and distribute information
to support decision making and control in an organisation. They identify five
components of information systems: hardware, software, data, people, and
processes.

Web-based information systems extend traditional MIS capabilities by making
information accessible over the internet through standard web browsers,
eliminating the need for specialised client software and enabling access from
any location (O'Brien & Marakas, 2011). This characteristic makes web-based
systems particularly suitable for student-facing applications in universities,
where users access the system from diverse devices and locations.

The application of MIS to hostel management is a specific instance of what
Turban et al. (2015) describe as operational information systems — systems
designed to support day-to-day operational activities including record keeping,
transaction processing, and routine reporting.


2.3 HOSTEL MANAGEMENT SYSTEMS — GLOBAL PERSPECTIVE
----------------------------------------------------

The automation of student accommodation management has been widely studied
and implemented across universities globally. Davis (2019) notes that
universities in the United Kingdom, Australia, and the United States have
largely transitioned to fully integrated online accommodation portals that
allow students to apply for accommodation, rank preferences, receive offers,
sign digital tenancy agreements, and manage payments entirely online.

Anderson and Thompson (2020) conducted a comparative analysis of hostel
management systems across fifteen universities in the UK and Australia,
finding that institutions with fully automated accommodation management
systems reported a 67% reduction in administrative processing time and a
45% decrease in room allocation errors compared to those still using manual
or semi-automated approaches.

In Asia, Singh and Kumar (2018) developed a web-based Hostel Management System
for an engineering college in India using PHP and MySQL, demonstrating the
feasibility of automating allocation, fee management, and complaint handling
in a single integrated platform. Their system reduced the time taken for room
allocation from an average of three days to under two hours.

Zhang et al. (2019) proposed a smart hostel management system incorporating
IoT sensors for occupancy detection, arguing that the combination of web-based
management with real-time physical monitoring represents the next generation
of accommodation management. While such advanced integration is beyond the
scope of this study, it highlights the trajectory of the field.


2.4 HOSTEL MANAGEMENT IN AFRICAN UNIVERSITIES
-----------------------------------------------

Research on hostel management systems in Sub-Saharan African universities
reveals a significant digital divide compared to Western counterparts.
Asante and Boateng (2021) surveyed twenty universities in Ghana, Nigeria,
and Uganda and found that over 70% still rely primarily on manual or
spreadsheet-based systems for hostel management. They attribute this to
limited IT infrastructure investment, lack of ICT policy support, and
the absence of locally developed systems tailored to the specific needs
and constraints of African institutional contexts.

Osei and Mensah (2020) developed an online student accommodation management
system for a Ghanaian polytechnic using PHP and MySQL. Their study identified
the same core challenges found in this study — information inaccessibility,
slow allocation, and poor payment tracking — and demonstrated that a
web-based system could address these effectively within the budget and
infrastructure constraints typical of Ghanaian polytechnics.

In Uganda, Ssemugabi and Masika (2019) analysed hostel management practices
at three Kampala-based universities and documented significant student
dissatisfaction arising from lack of transparency in room allocation,
long waiting times at hostel offices, and frequent disputes over payment
records. They recommended the adoption of web-based hostel management systems
as a priority ICT investment for Ugandan universities.

Kato and Nakayama (2022) specifically studied the hostel management system at
Makerere University, the largest public university in Uganda, noting that
while the university had attempted partial digitisation through spreadsheets,
the absence of a fully integrated system continued to cause operational
inefficiencies. Their study concluded that a purpose-built web application
would yield significant operational benefits.


2.5 EXISTING RELATED SYSTEMS
------------------------------

Several existing hostel management systems were reviewed to inform the design
of this system:

(a) eHMS (Electronic Hostel Management System) — India
This system, documented by Patel et al. (2020), is a PHP-based web application
used by several Indian engineering colleges. It supports student registration,
room allocation, fee payment, and complaint management. Strengths include a
comprehensive feature set; weaknesses include a dated interface, lack of mobile
responsiveness, and no support for file upload (payment receipts).

(b) UniAccomm — United Kingdom
UniAccomm is a commercial accommodation management platform used by several
UK universities. It provides a sophisticated end-to-end solution including
online applications, digital tenancy agreements, room inventory management,
and payment processing. However, it is a commercial product with high
licensing costs and is designed for the UK context — making it unsuitable
for direct adoption by Ugandan universities.

(c) HMS Pro — Nigeria
Documented by Adeyemi and Obi (2021), HMS Pro is a Java-based desktop
application developed for a Nigerian federal university. While functional,
its desktop-based architecture limits accessibility and requires installation
on each workstation — a significant constraint in resource-limited environments.

(d) Open Source Solutions (Django/PHP Templates)
Several open-source templates and demo projects for hostel management built
on Django and PHP were reviewed on platforms like GitHub. These provided
useful reference implementations for data modelling and workflow design but
lack the specific feature set, customisation, and security hardening required
for this institution.


2.6 GAPS IN EXISTING LITERATURE
---------------------------------

From the review of existing literature and systems, the following gaps are
identified that this study seeks to address:

(i) Context Specificity: Most existing systems are designed for Indian,
    UK, or Nigerian contexts. There is limited documented work on hostel
    management systems specifically developed for Ugandan private universities,
    where the specific workflow (manual payment submission, bank transfer
    verification, academic year segmentation) differs from other contexts.

(ii) Booking Request Workflow: Existing systems reviewed either rely on
     fully automated allocation algorithms or simple waiting lists.
     None provides a structured booking request workflow that allows
     students to express hostel preferences which administrators then act
     upon — a workflow better suited to the manual-allocation context of
     Ugandan institutions.

(iii) Receipt Printing: None of the reviewed systems provides a dedicated
      browser-based printable receipt that combines payment details, allocation
      information, and student profile in a single professional document.

(iv) Profile Photo Integration: Student identity management through profile
     photo upload is not addressed in the reviewed systems, yet it is
     important for physical verification purposes in the Ugandan context.

(v) Open Source, Locally Deployable Solution: There is a clear need for
    a freely available, locally deployable solution built on modern web
    technology that can be adapted by Ugandan universities without
    commercial licensing costs.


2.7 THEORETICAL FRAMEWORK
---------------------------

This study is grounded in the following theoretical frameworks:

(a) Systems Theory (Von Bertalanffy, 1968): The hostel management process is
    conceived as a system with defined inputs (student data, room data,
    payment data), processes (registration, allocation, verification,
    reporting), and outputs (allocation confirmations, receipts, reports).
    Systems theory provides the conceptual basis for the input-process-output
    model that underpins the system design.

(b) Technology Acceptance Model — TAM (Davis, 1989): TAM posits that users
    accept a technology based on its perceived usefulness and perceived ease
    of use. This framework informed the user interface design decisions,
    particularly the emphasis on a clean, intuitive Bootstrap 5 interface
    and clear navigation — ensuring that both students and hostel managers
    can effectively adopt and use the system with minimal training.

(c) Model-View-Template (MVT) Architecture: The Django framework's MVT
    pattern — a variant of the classic Model-View-Controller (MVC) pattern —
    provides the architectural foundation for the system. This pattern
    separates concerns between data modelling (Models), business logic
    (Views), and presentation (Templates), enabling maintainable and
    extensible code.

(d) Role-Based Access Control (RBAC) Theory (Ferraiolo et al., 2001):
    RBAC is applied to enforce differential access rights for the two
    primary user roles (Student and Hostel Manager), ensuring that each
    user can only access functions appropriate to their role.

--------------------

This study employed an applied research design — specifically a software
development research approach. Applied research focuses on solving a specific,
practical problem through the systematic development and testing of a solution
(Creswell, 2014). The problem being addressed is the inefficiency of the manual
hostel management process at Cavendish University Uganda, and the solution is
a purpose-built web application.

The development was guided by requirements elicited from primary and secondary
sources — including interviews with hostel staff, observation of the current
process, and review of existing manual records. These requirements were
formalised into functional and non-functional specifications that governed
the design and development phases.


3.3 AGILE DEVELOPMENT METHODOLOGY
-----------------------------------

The Agile software development methodology, specifically the Scrum framework,
was adopted for this project. Agile is an iterative, incremental approach to
software development that prioritises working software, customer collaboration,
and adaptability to change (Beck et al., 2001 — Agile Manifesto).

3.3.1 Rationale for Choosing Agile
Agile was selected over traditional waterfall approaches for the following
reasons:

(i)  The requirements for the system were not fully defined at the outset
     and evolved as the developer gained a deeper understanding of the
     hostel management process through ongoing research and supervisor
     feedback.

(ii) Agile's iterative sprints allowed for regular review and course
     correction, which was particularly valuable given that this was a
     solo development project with a single developer acting as both
     developer and product owner.

(iii) Working software was produced at the end of each sprint, enabling
      progressive demonstration and feedback from the university supervisor
      and potential users.

3.3.2 Sprint Breakdown
The project was organised into three two-week sprints:

SPRINT 1 — Foundation (Weeks 1-2):
  - Requirements analysis and system design
  - Database schema design and Django project setup
  - Custom user model implementation (users app)
  - Hostel, Block, Room models and Django Admin configuration
  - Base template and Bootstrap 5 design system
  - User registration and login functionality

SPRINT 2 — Core Features (Weeks 3-4):
  - Room allocation module (create, list, vacate)
  - Booking request workflow (student submission, admin review)
  - Payment submission and verification module
  - Role-based access control (decorators: @manager_required, @student_required)
  - Reports module (occupancy, allocation, payment reports with CSV export)
  - Student and admin dashboards

SPRINT 3 — Enhancement and Polish (Weeks 5-6):
  - Profile photo upload (ImageField, media file handling)
  - Printable payment receipt (dedicated view with @media print CSS)
  - Public hostel page with room type pricing and facilities
  - Hostel image display (image_url field)
  - N+1 query optimisation (prefetch_related for report views)
  - Context processor for sidebar badges
  - Final bug fixes, UI polish, and system testing

3.4.1 Django Framework
Django is a high-level Python web framework that encourages rapid development
and clean, pragmatic design. It follows the Model-View-Template (MVT)
architectural pattern and includes built-in features for URL routing,
database abstraction via ORM, template rendering, form handling, user
authentication, and administration. Django's philosophy of "batteries
included" made it particularly suitable for this project, as it reduced
the amount of boilerplate code required and provided robust built-in security
features including CSRF protection, SQL injection prevention, and
session management.

3.4.2 Bootstrap 5.3
Bootstrap is the world's most popular CSS framework for building responsive,
mobile-first web interfaces. Bootstrap 5.3 was selected for its clean
component library, excellent grid system, and CDN availability — eliminating
the need to host CSS assets locally during development. Custom CSS variables
and a custom design system (style.css) were layered on top of Bootstrap
to achieve a distinctive institutional visual identity.

3.4.3 SQLite Database
SQLite was chosen as the database for development due to its zero-configuration
nature and seamless integration with Django. It is file-based (stored as
db.sqlite3) and requires no separate database server process. While SQLite
is not recommended for high-concurrency production environments, it is
entirely appropriate for the development and testing phases of this project.
Django's ORM abstracts the database layer, making migration to PostgreSQL
straightforward if production deployment is required.


3.5 DATA COLLECTION METHODS
-----------------------------

Data for requirements analysis was collected through the following methods:

(i) Interview: A structured interview was conducted with the hostel
    administrator at Cavendish University Uganda to understand the current
    manual process, pain points, and requirements for an automated system.
    The interview covered the allocation workflow, payment procedures,
    reporting needs, and the types of users who interact with the system.

(ii) Observation: The researcher observed the manual hostel management
     process at the hostel office, documenting the sequence of activities
     from student inquiry through allocation and payment to move-in.
     This provided direct insight into inefficiencies and bottlenecks.

(iii) Document Review: Existing hostel registration forms, payment receipt
      templates, room register ledgers, and occupancy reports were reviewed
      to understand the data entities, fields, and relationships that the
      system needed to model.

(iv) Literature Review: Secondary data was collected from academic journals,
     textbooks, and online resources (as documented in Chapter 2) to
     benchmark the system against existing solutions and best practices.

3.6.1 Functional Requirements
The following functional requirements were identified through requirements
analysis:

Table 3.3: Functional Requirements

FR01 — The system shall allow new students to register accounts with their
        personal details including student ID, department, gender, and
        contact information.

FR02 — The system shall authenticate registered users through a username
        and password login mechanism and redirect them to a role-appropriate
        dashboard.

FR03 — The system shall display a public hostel listing page showing all
        hostels with their name, gender type, available facilities, room
        types, and prices per semester — accessible without login.

FR04 — Authenticated students shall be able to browse hostel listings with
        room availability information and submit a booking request for a
        preferred hostel.

FR05 — Students shall be able to cancel a pending booking request.

FR06 — Hostel managers shall be able to view all pending booking requests
        and use them to inform allocation decisions.

FR07 — Hostel managers shall be able to create room allocations assigning
        a specific student to a specific room for an academic year.

FR08 — The system shall prevent a room from being over-allocated beyond its
        stated capacity.

FR09 — The system shall prevent a student from having more than one active
        room allocation at a time.

FR10 — Hostel managers shall be able to vacate (mark as 'left') an active
        allocation when a student moves out.

FR11 — Students shall be able to submit a payment record including the
        payment amount, reference number, payment method, and an optional
        proof of payment file.

FR12 — Hostel managers shall be able to verify or reject a submitted payment
        record, with the verification recorded including the verifying
        administrator and timestamp.

FR13 — Authenticated students shall be able to view and print a detailed
        payment receipt combining payment details, allocation information,
        and student profile.

FR14 — Students shall be able to upload and update a profile photo.

FR15 — The system shall generate an occupancy report showing total capacity,
        occupied beds, and occupancy percentage per hostel.

FR16 — The system shall generate allocation and payment reports filterable
        by academic year and exportable to CSV format.

FR17 — Hostel managers shall be able to create, edit, and manage hostel,
        block, and room records through a web interface.

3.6.2 Non-Functional Requirements

Table 3.4: Non-Functional Requirements

NFR01 — Security: The system shall implement CSRF protection on all POST
         forms, session-based authentication, and role-based access control.
         The logout function shall use POST-only requests to prevent
         CSRF logout attacks.

NFR02 — Usability: The interface shall be responsive and usable on desktop
         and mobile devices. Navigation shall be clear and intuitive,
         with role-appropriate sidebars.

NFR03 — Performance: List views shall be paginated (20 records per page)
         to prevent slow page loads with large datasets. Report views shall
         use SQL aggregation rather than Python-level computation.

NFR04 — Reliability: The system shall enforce data integrity through both
         application-level validation (Django model clean() methods) and
         database-level constraints (unique constraints, foreign keys).

NFR05 — Maintainability: The system shall follow Django's MVT pattern with
         clearly separated apps for each domain module. Code shall be
         readable and documented.

NFR06 — Scalability: The database abstraction layer (Django ORM) shall
         allow migration to a production-grade database (PostgreSQL) with
         minimal code changes.

4.3 SYSTEM ARCHITECTURE
-------------------------

The system follows a three-tier architecture:

[See Figure 4.2: System Architecture Diagram — refer to diagram file
02_system_architecture.drawio]

TIER 1 — PRESENTATION LAYER:
The presentation layer consists of the user interface delivered to the web
browser. It comprises Bootstrap 5.3 CSS framework for responsive layout and
styling, Bootstrap Icons for iconography, Chart.js for dashboard charts,
a custom CSS design system (static/css/style.css), and Django's template
engine (Jinja2-based) for server-side HTML rendering. The layer is organised
around a master base.html template that implements two distinct layouts:
an authenticated sidebar layout for logged-in users, and a public navigation
layout for unauthenticated visitors.

TIER 2 — APPLICATION / BUSINESS LOGIC LAYER:
The application layer implements all business logic through Django's MVT
pattern. It is divided into five domain-specific Django applications:

- users: Handles authentication (registration, login, logout), role-based
  dashboards, user profiles, and profile photo management.

- hostels: Manages hostel, block, and room entities including CRUD views,
  the public hostel listing page with pricing, and the admin hostel
  management interface.

- allocations: Implements room allocation creation, listing, and vacating,
  as well as the booking request workflow for students and managers.

- payments: Handles payment record submission by students, payment
  verification by managers, and printable receipt generation.

- reports: Generates occupancy, allocation, and payment reports with
  academic year filtering and CSV export.

Cross-cutting concerns handled at the application layer include:
- CSRF protection (Django middleware)
- Session-based authentication (Django auth)
- Role-based access control via custom decorators (@manager_required,
  @student_required in users/decorators.py)
- Context processors for sidebar badge counts (users/context_processors.py)

TIER 3 — DATA LAYER:
The data layer consists of the Django ORM providing an abstraction over
the underlying SQLite database. Models define the schema, relationships,
and constraints. Django migrations manage schema versioning. Media files
(profile photos, payment receipts) are stored in the filesystem under the
MEDIA_ROOT directory.

4.6 LEVEL 1 DATA FLOW DIAGRAM
-------------------------------

[See Figure 4.5: Level 1 DFD — refer to diagram file
06_dfd_level1.drawio]

The Level 1 DFD decomposes the central HMS process into six sub-processes:

Process 1.0 — User Registration and Authentication:
Receives registration details from the Student and stores user records
in the User Store (D1). On subsequent visits, retrieves credentials from
D1 to authenticate users and issue login confirmations.

Process 2.0 — Hostel and Room Management:
Receives hostel and room data from the System Admin, stores it in the
Hostel/Room Store (D2), and retrieves it for display on public and
authenticated hostel listing pages.

Process 3.0 — Booking Request Management:
Receives booking request details from the Student, saves them to the
Booking Request Store (D5), and returns booking status to the Student.

Process 4.0 — Room Allocation Management:
Receives allocation data from the Hostel Manager, retrieves available
room data from D2, stores allocations in the Allocation Store (D3),
and returns allocation confirmations.

Process 5.0 — Payment Processing and Verification:
Receives payment details and receipt files from the Student and
verification decisions from the Manager. Stores payment records in the
Payment Store (D4) and returns payment receipts to the Student.

Process 6.0 — Report Generation:
Reads data from D2, D3, and D4 to compile occupancy, allocation, and
payment reports, which are delivered to the Hostel Manager in HTML or
CSV format.

Data Stores:
D1 — User Store (users_customuser table)
D2 — Hostel/Room Store (hostels_hostel, hostels_block, hostels_room tables)
D3 — Allocation Store (allocations_allocation table)
D4 — Payment Store (payments_payment table)
D5 — Booking Request Store (allocations_bookingrequest table)


4.7 ENTITY RELATIONSHIP DIAGRAM
---------------------------------

[See Figure 4.6: ERD — refer to diagram file 07_erd.drawio]

The system database comprises seven primary entities with the following
relationships:

Entities and Relationships:

CustomUser — the central entity representing all system users. It has
one-to-many relationships with Allocation (a student can have multiple
allocations over time), Payment (a student can make multiple payments),
and BookingRequest (a student can make multiple booking requests over
different academic years). A manager (also a CustomUser) can verify
multiple Payments (captured by the verified_by FK).

Hostel — represents a hall of residence. One Hostel contains many Blocks
(1:N relationship). One Hostel can receive many BookingRequests from
students (1:N).

Block — represents a wing or section within a hostel. One Block contains
many Rooms (1:N relationship). A Block belongs to exactly one Hostel.

Room — represents an individual room within a Block. One Room can have
many Allocations over time (1:N), as successive students may occupy the
same room across different academic years. A Room belongs to exactly
one Block.

Allocation — the linking entity between CustomUser (student) and Room.
It records when a student is assigned to a room, the academic year, and
the current status (active/left/cancelled). A unique constraint enforces
that at most one active allocation exists per student at any time.

Payment — records a payment submitted by a student. It links to the
submitting CustomUser (user FK) and to the verifying manager
(verified_by FK). Each payment has a unique reference_number.

BookingRequest — records a student's expressed preference for a hostel.
It links to the requesting student (student FK) and the preferred hostel
(preferred_hostel FK).

4.9 USER INTERFACE DESIGN
---------------------------

The system implements two distinct layouts within a single base template:

Public Layout (unauthenticated users):

Registration:
New students register through the /register/ URL. The StudentRegistrationForm
collects username, first_name, last_name, email, student_id, and password.
On successful registration, is_student is automatically set to True and the
user is redirected to the login page.

Authentication:
Login is handled by Django's AuthenticationForm at /login/. Successful
authentication redirects the user to /dashboard/, which dispatches to the
role-appropriate dashboard (admin_dashboard or student_dashboard) based on
the user's role flags.

Logout:
The logout view is POST-only, protecting against CSRF logout attacks where
an attacker could embed a malicious URL (e.g., in an image tag) to forcibly
log out authenticated users.

Role-Based Access Control:
Two custom decorators enforce role-based access:
- @manager_required: Redirects unauthenticated users to login and
  non-manager users to the dashboard with an error message.
- @student_required: Redirects unauthenticated users to login and
  manager users to the admin dashboard.


5.4 HOSTEL AND ROOM MANAGEMENT
--------------------------------

The hostels app manages the three-level physical hierarchy:
Hostel → Block → Room

Hostel Model Fields:
name (unique), location, gender (Male/Female/Mixed), description,
image_url (URL to banner photo), price_per_semester, created_at, updated_at.

Computed methods on Hostel:
- total_capacity(): Sum of all room capacities across all blocks
- available_spaces(): total_capacity - occupied_beds
- occupied_beds(): Sum of active allocations across all rooms

Block Model Fields:
hostel (FK), name, description, created_at.
The capacity field is a computed @property (sum of room capacities)
rather than a stored database field, eliminating the risk of desynchronisation.

Room Model Fields:
block (FK), room_number, capacity (beds), room_type (Single/Double/
Self-Contained/Shared), price_per_semester, description, created_at.
The is_full field is a computed @property rather than a stored boolean,
preventing stale state.
The occupied_beds() method uses self.allocations.all() which works with
Django's prefetch_related to avoid N+1 queries in list views and reports.

Web Interface:
The hostels app provides authenticated CRUD views for managers (hostel_list,
hostel_detail, hostel_create, hostel_edit, block_create, room_create).
A separate public_hostels view renders the unauthenticated public hostel page.
An authenticated student_hostels view in the allocations app renders the
hostel browsing page for logged-in students, with booking buttons.

Public Hostel Page Features:
- Banner photo with overlay (from image_url field)
- Available Facilities: Security, Free Water, Free Electricity, Free WiFi
- Room types table showing total beds, available beds, and price per type
- Book This Hostel button (for logged-in students without existing allocation)


5.5 BOOKING REQUEST MANAGEMENT
--------------------------------

The booking request workflow is implemented in the allocations app using
the BookingRequest model, which records a student's expressed preference
for a hostel before room assignment.

Student Workflow:
1. Student browses the authenticated hostel listing (/allocations/browse-hostels/)
2. Student clicks "Book This Hostel" on a preferred hostel
3. Student fills in the booking request form (academic year, optional message)
4. System validates: no existing active allocation, no existing pending/approved
   request
5. Booking request saved with status='pending'
6. Student can cancel a pending request from the dashboard

Manager Workflow:
1. Manager views all booking requests at /allocations/booking-requests/
   with filter tabs (Pending, Approved, Rejected, Cancelled)
2. Manager uses the pending request information to inform allocation decisions
3. When a room allocation is created for a student, any pending booking
   requests for that student are automatically marked as 'approved'

Business Rules Enforced:
- A student cannot have more than one active (pending or approved) booking
  request at a time
- A student with an active room allocation cannot submit a new booking request
- Only pending requests can be cancelled by the student


5.6 ROOM ALLOCATION MANAGEMENT
--------------------------------

Room allocation is implemented in the allocations app using the Allocation model.

Allocation Model Fields:
user (FK to CustomUser), room (FK to Room), status (active/left/cancelled),
date_allocated (auto), date_left (nullable), academic_year, notes.

Capacity Validation:
The Allocation model's clean() method validates:
(i)  Room not over-capacity: active allocation count < room.capacity
(ii) No dual active allocation: no other active allocation exists for
     the same student
(iii) Gender matching: male students cannot be allocated to Female-only
      hostels and vice versa
The clean() method is called from save() to enforce validation on every save.
A database-level UniqueConstraint provides additional protection against
concurrent allocation creation.

Manager Workflow:
1. Manager navigates to New Allocation (/allocations/create/)
   - If accessed from the Booking Requests page with ?student=<pk>,
     the student field is pre-populated automatically
2. Manager selects student and room from dropdowns
3. System runs validation; if valid, allocation is created
4. Any pending booking request for the student is automatically approved

Vacate Room:
A manager can mark an active allocation as 'left' through the Vacate action
on the allocation list. This sets status='left' and records date_left.

Allocation List:
The allocation list view supports search (by student name, student ID, room
number, hostel name) and status filtering. Results are paginated at 20
records per page.


5.7 PAYMENT MANAGEMENT
------------------------

The payment module is implemented in the payments app.

Payment Model Fields:
user (FK), amount (Decimal), reference_number (unique, mandatory),
status (pending/verified/rejected), date_paid (auto), date_verified,
verified_by (FK to manager, nullable), payment_method, academic_year,
notes, receipt (FileField for proof of payment).

Student Submission:
Students access the payment form at /payments/create/ (student-only,
guarded by @student_required). They enter the payment amount, reference
number (bank/mobile money transaction ID), payment method, academic year,
optional notes, and optionally upload a proof of payment file.
The payment.user is set automatically to request.user on save.
The reference_number unique constraint prevents duplicate payment records.

Manager Verification:
The payment list at /payments/ (manager-only) shows all payments paginated.
Each payment row has a "Verify" button linking to /payments/verify/<id>/.
The verify view allows the manager to:
- Mark as Verified: sets status='verified', records verified_by and
  date_verified
- Mark as Rejected: sets status='rejected'

Payment Receipt:
The payment receipt view at /payments/receipt/<id>/ is accessible by the
owning student or any manager. It renders a printable HTML page including:
- University header
- Payment status banner (colour-coded: green/orange/red)
- Student photo (or initials fallback) and personal details
- Payment details table
- Room allocation section (if allocated)
- Official stamp and signature area
- @media print CSS hides sidebar and navbar, printing only the receipt

On-Screen Action:
Students access the receipt from their dashboard payment history.
Managers access it from the payment list.
A "Print Receipt" button triggers window.print() for browser-based printing.


5.8 REPORT GENERATION
-----------------------

The reports app provides three report views, all protected by @manager_required:

(i) Hostel Occupancy Report (/reports/occupancy/):
Displays per-hostel statistics: total capacity, occupied beds, available beds,
and occupancy percentage (colour-coded progress bar). Uses prefetch_related(
'blocks__rooms__allocations') to load all data in a single database query
set, with occupied_beds() method using prefetch cache to avoid N+1 queries.

(ii) Allocation Report (/reports/allocations/):
Displays all allocations with student name, student ID, hostel, block, room,
status, date, and academic year. Supports filtering by academic year (dropdown).
Exportable to CSV (format=csv query parameter).

(iii) Payment Report (/reports/payments/):
Displays all payments with summary statistics (total, verified, pending amounts)
using SQL aggregation (aggregate(Sum('amount'))). Supports academic year
filtering. Exportable to CSV.

CSV Export:
Both allocation and payment reports support CSV export via Django's
HttpResponse with content_type='text/csv' and Content-Disposition attachment
header. CSV files are generated server-side without any additional libraries.


5.9 SECURITY IMPLEMENTATION
-----------------------------

The following security measures are implemented in the system:

(i) CSRF Protection: Django's CsrfViewMiddleware is enabled, adding CSRF
    tokens to all POST forms. The logout form uses POST to prevent CSRF
    logout attacks.

(ii) Session Authentication: Django's session-based authentication ensures
     that user identity is verified on every request. The @login_required
     decorator protects all non-public views.

(iii) Role-Based Access Control: Custom decorators (@manager_required and
      @student_required) enforce role-appropriate access. Both decorators
      check is_authenticated first to handle unauthenticated requests safely.

(iv) Password Security: Django's AUTH_PASSWORD_VALIDATORS enforce minimum
     length, common password detection, and numeric-only password prevention.
     Passwords are stored as secure hashes using Django's PBKDF2-SHA256
     algorithm.

(v) File Upload Security: Payment receipts and profile photos are uploaded
    to designated subdirectories within MEDIA_ROOT. Django does not serve
    uploaded files with execute permissions.

(vi) Secret Key Management: The Django SECRET_KEY is read from an environment
     variable (DJANGO_SECRET_KEY) with a fallback for local development only.
     This prevents accidental exposure of secret keys in source code.

(vii) Input Validation: All form inputs are validated through Django's form
      validation layer before being saved to the database. Model-level
      clean() methods provide additional business rule validation.

that the system meets its specified requirements and behaves as expected under

      collecting structured feedback on usability, completeness, and
      correctness.

Django's built-in development server (python manage.py runserver) was used as
the test environment. The Django system check framework (python manage.py check)
was run after every significant change to verify the integrity of the
application configuration. All 20 database migrations were applied cleanly

7.1 CONCLUSION
---------------

This project has successfully designed and developed an Online Hostel
Information Management System for Cavendish University Uganda. The system
was built using the Python Django web framework, Bootstrap 5.3 for the
frontend, and SQLite as the development database.

The system addresses all five problems identified in the problem statement:

(i)  Information Inaccessibility: Resolved through a public hostel listing
     page with hostel images, room type categories, pricing, and available
     facilities — accessible without login from any web browser.

(ii) Slow and Error-Prone Allocation: Resolved through a structured digital
     allocation workflow with automatic capacity validation, gender matching,
     and dual-allocation prevention enforced at both the application and
     database levels.

(iii) Poor Payment Tracking: Resolved through a digital payment submission
      workflow with proof-of-payment file upload, manager verification with
      complete audit trail (verified_by, date_verified), and printable
      payment receipts.

(iv) Lack of Reporting Capability: Resolved through three real-time report
     views (occupancy, allocation, payment) with academic year filtering,
     CSV export, and performance-optimised SQL aggregation queries.

(v)  No Booking Request Workflow: Resolved through a structured booking
     request module allowing students to express hostel preferences,
     with manager review and automatic approval on allocation creation.

The Agile Scrum methodology proved effective for this project, enabling
progressive development and refinement across three two-week sprints.
All 17 functional requirements and 6 non-functional requirements identified
during analysis were implemented and verified through testing.


7.2 ACHIEVEMENTS
-----------------

The key achievements of this project are:

1. A fully functional, role-based web application with two user roles
   (Student and Hostel Manager) and a system administrator via Django Admin.

2. A complete hostel inventory management system with three-level hierarchy
   (Hostel → Block → Room) including room type categorisation and pricing.

3. A structured booking request workflow that bridges student preferences
   with manager allocation decisions.

4. A secure payment management module with file upload capability, manager
   verification workflow, and professional printable receipts.

5. A performance-optimised reporting module with CSV export capability.

6. A responsive, modern user interface built on Bootstrap 5.3 with a
   sidebar navigation layout and role-appropriate content.

7. Strong security implementation including CSRF protection, POST-only
   logout, role-based access control, and environment-based secret key
   management.

8. A comprehensive architectural documentation (CLAUDE.md) that serves
   as living reference for future developers maintaining the system.


7.3 LIMITATIONS OF THE SYSTEM
-------------------------------

Despite the achievements, the following limitations were identified:

(i)  No Email/SMS Notifications: Students receive no automatic notifications
     when their booking request is reviewed, room is allocated, or payment
     is verified. They must actively check the system.

(ii) No Automatic Payment Gateway: Payment submission is manual (students
     enter transaction reference numbers from bank/mobile money confirmations).
     There is no direct integration with MTN Mobile Money, Airtel Money,
     or bank payment APIs.

(iii) SQLite Database: While suitable for development and small-scale
      deployment, SQLite does not support high-concurrency production
      workloads. A race condition exists where two concurrent allocation
      requests could theoretically both pass validation simultaneously.

(iv) No Automated Allocation Algorithm: Room allocation is entirely manual.
     There is no option for automated matching of students to rooms based
     on preferences, gender, academic year, or other criteria.

(v)  Single Institution: The system is designed for a single institution.
     Multi-tenancy support for multiple universities would require
     significant architectural changes.

(vi) No Mobile App: The system is web-based only. While the interface is
     mobile-responsive, a dedicated Android/iOS app would provide a better
     mobile experience.


7.4 RECOMMENDATIONS FOR FUTURE WORK
-------------------------------------

Based on the limitations identified and feedback from User Acceptance Testing,
the following recommendations are made for future enhancement of the system:

1. Email and SMS Notifications: Integrate Django's email backend and a
   Ugandan SMS gateway (e.g., Africa's Talking, Yo! Uganda) to send automatic
   notifications on booking request status changes, allocation creation,
   and payment verification.

2. Payment Gateway Integration: Integrate with MTN Mobile Money and Airtel
   Money APIs to allow direct in-system payment processing, eliminating
   the need for manual reference number entry and reducing the risk of
   fraudulent payment claims.

3. PostgreSQL Migration: Migrate from SQLite to PostgreSQL for production
   deployment. PostgreSQL provides better concurrency handling, row-level
   locking, and support for advanced constraint features. Django's ORM
   makes this migration straightforward.

4. Automated Room Allocation: Implement an allocation algorithm that can
   automatically suggest or assign rooms based on student gender, booking
   request preferences, and academic year — with manager review and approval.

5. Student Maintenance Request Module: Add a maintenance/complaint reporting
   module allowing students to report room maintenance issues (broken
   fixtures, plumbing, electricity) with status tracking.

6. Enhanced Academic Year Management: Add a dedicated AcademicYear model
   with start/end dates, making it easier to archive old records and
   generate year-on-year comparative reports.

7. Mobile Application: Develop a companion Android application using
   Django REST Framework as the backend API, providing a native mobile
   experience for students.

8. Two-Factor Authentication: Add optional 2FA for hostel manager accounts
   to enhance security of privileged operations.

class Allocation(models.Model):
    user = models.ForeignKey(settings.AUTH_USER_MODEL,
                             on_delete=models.CASCADE,
                             related_name='allocations')
    room = models.ForeignKey(Room, on_delete=models.CASCADE,
                             related_name='allocations')
    status = models.CharField(max_length=20, choices=STATUS_CHOICES,
                              default='active')
    date_allocated = models.DateTimeField(auto_now_add=True)

    def clean(self):
        # Prevent over-capacity
        active_count = Allocation.objects.filter(
            room=self.room, status='active'
        ).exclude(pk=self.pk).count()
        if active_count >= self.room.capacity:
            raise ValidationError(f'Room {self.room.room_number} is full.')
        # Prevent dual active allocation
        if Allocation.objects.filter(
            user=self.user, status='active'
        ).exclude(pk=self.pk).exists():
            raise ValidationError(
                f'{self.user} already has an active room allocation.')
        # Gender matching
        hostel_gender = self.room.block.hostel.gender
        user_gender = self.user.gender
        if hostel_gender == 'Male' and user_gender == 'Female':
            raise ValidationError('Female students cannot be allocated to a
                                   Male hostel.')
        if hostel_gender == 'Female' and user_gender == 'Male':
            raise ValidationError('Male students cannot be allocated to a
                                   Female hostel.')

B.3: Role-Based Access Decorators (users/decorators.py)
---------------------------------------------------------

from functools import wraps
from django.shortcuts import redirect
from django.contrib import messages

def manager_required(view_func):
    @wraps(view_func)
    def wrapper(request, *args, **kwargs):
        if not request.user.is_authenticated:
            return redirect('login')
        if request.user.is_manager or request.user.is_staff:
            return view_func(request, *args, **kwargs)
        messages.error(request,
                       'You do not have permission to access this page.')
        return redirect('dashboard')
    return wrapper

def student_required(view_func):
    @wraps(view_func)
    def wrapper(request, *args, **kwargs):
        if not request.user.is_authenticated:
            return redirect('login')
        if request.user.is_manager or request.user.is_staff:
            return redirect('admin_dashboard')
        return view_func(request, *args, **kwargs)
    return wrapper

14. Is there anything else about the current hostel management process that
    you think I should know?
